The Problem

What pharma reps deal with every day

❌ Who do I contact next?

Reps manage 100+ HCPs with no clear signal on who needs attention right now. Priority is often based on gut feel or last CRM login date.

❌ What channel should I use?

Email to someone who never opens it. Calls to someone who never picks up. Reps waste time on the wrong channel because there's no intelligent routing.

❌ Am I even allowed to contact?

Opt-outs, adverse event holds, frequency caps, blackout periods — compliance is manual, error-prone, and costly when breached.

❌ What content do I send?

Approved content libraries have 50+ assets. Reps default to generic emails rather than tailoring content to the HCP's therapy area and engagement history.

❌ Why am I recommending this?

When challenged by a manager or compliance team, reps can't articulate why they chose a specific action. No audit trail. No transparent rationale.

❌ When is the right moment?

Too soon after a meeting? HCP is irritated. Too late after a webinar? The engagement momentum is lost. Timing is everything — and nobody's tracking it.

Our Solution

An AI Copilot that answers all of this — automatically

A lightweight agentic system that assembles an HCP context view, runs compliance guardrails, retrieves approved content, and uses an LLM to produce a clinically-grounded, ready-to-act recommendation.

🛡️

Guardrails First

Rules engine runs before any AI. Hard blocks (opt-out, AE hold) are deterministic — no LLM can override them.

📚

Approved Content Only

Content retrieval hard-filters on approval status and therapy area match. The LLM cannot hallucinate assets that don't exist.

🧠

LLM Reasoning

Llama 3.3 70B synthesises HCP profile, engagement signals, rules output, and content to produce a specific, traceable recommendation.

✍️

Draft Communication

Generates a ready-to-send email, structured call plan, or meeting prep note — personalised to the HCP.

🎯

Core Design Principle

This is a reasoning and workflow tool, not a prediction engine. The rep stays in control — the Copilot provides structured intelligence so every decision is faster, safer, and better explained.

Key Design Decisions

  • Rules run before LLM — compliance cannot be hallucinated
  • LLM only sees pre-filtered approved content
  • Temperature 0.3 — deterministic, clinically consistent
  • JSON output enforced — no free-text drift
  • Stateless API — each call is independent

Guardrails

Rules engine — how compliance is enforced

Runs in strict priority order. First matching hard block terminates immediately — no LLM call is made.

1

Opt-Out Check — HARD BLOCK

If hcp.optOut == true → return "Do Not Contact". Unsubscribed HCPs are never reachable via this system.

2

Compliance Blackout — HARD BLOCK

If hcp.blackout == true → return blocked reason (e.g. adverse event hold). Medical affairs sets this flag manually.

3

Minimum Frequency Gap — HARD BLOCK

If contacted < 3 days ago → blocked. Prevents aggressive re-contact that would violate industry norms.

4

Recent Contact Warning — SOFT WARNING

If contacted < 7 days ago → warning passed to LLM. Allows contact but flags it; LLM may recommend lighter touch.

5

Channel Preference Logic

Prefers email but 0 opens? → switch to phone/f2f. Prefers f2f but last was f2f <21 days? → lighter touch. Webinar preference? → recommend webinar + email.

6

HCP Status Classification

New (no history) · Active (<45 days) · Re-Engaging (45–90 days) · Churned (>90 days). Status shapes LLM timing guidance.

Engagement Scoring (0–10)

Email opens (max 10)× 0.40
Webinar attendance (max 5)× 0.80
Portal logins (max 15)× 0.20
Click-throughs (max 3)× 0.50
Total (capped at 10)0–10

Why Rules First?

In pharma, compliance is non-negotiable. An LLM can hallucinate a justification for contacting an opted-out HCP if given the chance. By running rules deterministically before the LLM, we guarantee no AI reasoning can override a legal or safety constraint.

Tradeoffs & Design Decisions

Deliberate choices — and why we made them

JSON files vs. database

Chosen: Static JSON. Zero setup, portable, shareable. Trade-off: No persistence, no multi-user, not scalable. Appropriate for a prototype — swap to Postgres in production.

Rules engine vs. full ML model

Chosen: Deterministic rules. Auditable, debuggable, no training data needed. Trade-off: Less nuanced than a trained model. Right choice for compliance-critical guardrails.

Score-based retrieval vs. embeddings

Chosen: Explicit relevance scores per therapy area. Transparent, controllable. Trade-off: Less flexible than semantic search. In production: add FAISS or pgvector embeddings.

Groq (Llama 3.3) vs. GPT-4

Chosen: Open-source model via Groq. Fast, free tier, OpenAI-compatible API. Trade-off: Slightly less reasoning quality than GPT-4. Acceptable for structured JSON output at temp 0.3.

Single-turn vs. multi-turn LLM

Chosen: Single prompt with full context. Simple, stateless, fast. Trade-off: No memory across sessions. Production would add session state and conversation history.

No authentication

Chosen: Open access for prototype demo. Trade-off: Not deployable in production. Auth, role-based access, and rep-level personalisation needed for real use.

What's synthetic vs. real

  • HCP profiles — synthetic, representative
  • Therapy areas — real (HIV, CV, Endo…)
  • Engagement signals — plausible mock data
  • Content assets — synthetic but realistic pharma
  • Business rules — based on real pharma standards
  • No real patient data used anywhere
  • No trained predictive model
  • No real CRM integration

Temperature: 0.3 — Why?

Clinical recommendations require consistency. High temperature creates variance between runs for the same HCP — unacceptable when a rep presents the recommendation to a manager. Low temperature gives traceable, reproducible output.

Next Steps

What production-ready would look like

The prototype demonstrates the core workflow. Scaling to production involves several well-defined engineering upgrades — none of which change the fundamental agent architecture.

Data & Storage

🗄️  Replace JSON with PostgreSQL — proper HCP records, versioning, audit trail
🔍  Add vector embeddings (pgvector / FAISS) for semantic content matching
📡  CRM integration (Veeva / Salesforce) for live HCP data sync
📊  Event streaming (Kafka / Kinesis) for real-time engagement signal ingestion

Agent Intelligence

🧠  Multi-turn memory — track recommendation history per rep session
📈  Feedback loop — rep accepts/rejects recommendation → fine-tune prompt
🎯  Predictive scoring — trained model for engagement propensity
🌍  Multi-language draft generation for global territories

Compliance & Governance

🛡️  Configurable rules engine — medical affairs can edit rules without code deploys
📋  Full audit trail — every recommendation logged with HCP state snapshot
✅  Content approval workflow — new assets go through approval before entering library
🔒  GDPR / HIPAA-compliant data handling and access control

Platform & UX

🔑  Auth + RBAC — rep login, manager view, medical affairs dashboard
📱  Mobile-optimised — reps often work on tablets in the field
📤  Direct CRM push — one-click to log recommendation as CRM activity

What stays the same
